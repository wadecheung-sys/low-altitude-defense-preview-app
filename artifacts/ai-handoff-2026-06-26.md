## 1. 本阶段背景

本阶段围绕低空防御指挥控制平台前端演示工程，持续进行了多轮页面结构调整、文案统一、详情页演示增强、导出模板、系统管理扩展等工作。

后期重点问题集中在：

1. “飞行记录详情”页面相关逻辑不稳定，曾多次引发页面空白、加载异常、路由失效。
2. “设备信息”页面进入后，系统框架顶部面包屑出现双层级重复。
3. 系统中部分中文文案与路由标题存在编码/乱码问题。
4. 在尝试修复面包屑重复与威胁等级文案时，因基础源码本身已存在编码异常，导致后续修补风险扩大。

## 2. 已明确的核心结论

### 2.1 GitHub 主分支并不是“中文文案正常”的稳定基线

已确认 `origin/main` 可以成功构建，但并不等于“文案正常”：

1. 多个路由标题字段已是乱码状态。
2. `ladRouteMeta.ts`、`permissionSeed.ts` 等基础文件内存在较多乱码文案。
3. 某些字符串甚至出现断裂风险，影响路由配置安全性。

因此：

- `origin/main` 是“可构建稳定版本”
- 但不是“可直接作为中文产品展示基线的版本”

### 2.2 当前面包屑重复问题的根因已定位

“设备信息”页面顶部展示为：

- `设备管理 / 设备信息 / 设备信息`

根因不是页面渲染两次，而是路由结构本身存在包装层：

- `/lad/device`
- `/lad/device/info`
- `/lad/device/info/list`

其中：

1. `LadDevice` 标题为“设备管理”
2. `LadDeviceInfoWrap` 标题为“设备信息”
3. `LadDeviceInfoList` 标题也为“设备信息”

面包屑生成逻辑会将整条命中路径全部转为层级，因此出现重复。

### 2.3 本次错误修复方式的问题

曾尝试在共享面包屑过滤函数中，直接跳过 `LadDeviceInfoWrap`。

该方案的实际问题是：

1. 不是“隐藏一个中间面包屑节点”
2. 而是“直接跳过整个路由分支”
3. 导致其子节点的面包屑链路也失效

进而引发：

1. “设备信息”页面面包屑消失
2. “字典管理”依然重复
3. 其他使用包装路由的模块层级显示不一致

结论：

- 面包屑问题可以安全修复
- 但必须用“提升包装路由子节点”或“统一包装层策略”的方式处理
- 不能继续使用点状硬编码跳过某个路由名

## 5. 当前最重要的风险

### 5.1 基础文案层已经不可信

风险最高的不是某个业务页面，而是：

1. 路由标题
2. 页面元数据
3. 权限种子
4. 字典文案

这些“基础文案层”已经部分乱码。

继续在此基础上直接做功能迭代，极易导致：

1. 菜单、面包屑、标签页显示异常
2. 字符串断裂
3. 难以判断到底是逻辑问题还是编码问题

### 5.2 共享层补丁风险高于页面级补丁

本轮已经验证：

1. 修改共享面包屑函数
2. 修改路由包装层行为
3. 修改共享字典/等级映射

这类工作一旦建立在乱码基础上，风险会远高于单页面改动。

## 6. 对下一位接手者的建议

### 6.1 不建议马上继续修“设备信息面包屑”

原因：

1. 当前问题表面是设备信息重复
2. 实际底层是“包装路由 + 共享面包屑算法 + 乱码路由元信息”三者叠加
3. 如果不先建立统一规则，继续点修只会反复引发新问题

### 6.2 推荐优先顺序

建议按以下顺序推进。

#### 第一阶段：建立“中文基线”

目标：

- 找到一个菜单标题、路由元信息、权限名称都正常的版本

来源优先级建议：

1. 用户本地另一份能正常显示中文菜单的项目目录
2. GitHub 更早的某个 commit / tag / 分支
3. 历史可运行打包产物反向参考
4. 若以上均无，再人工重建基础文案层

需要优先核对的文件：

1. `src/router/ladRoutes.ts`
2. `src/router/ladRouteMeta.ts`
3. `src/api/lad/auth/permissionSeed.ts`
4. 系统字典与部分菜单展示相关文件

#### 第二阶段：统一“包装路由面包屑规则”

目标：

- 一次性解决“设备信息 / 字典管理 / 其他 getParentLayout 包装路由”的重复与缺失问题

推荐方式：

1. 不硬编码跳过单一路由名
2. 不直接删除包装层分支
3. 在面包屑树构建中识别“包装路由”
4. 对包装路由采用“子节点上提但保留链路”的统一策略

#### 第三阶段：逐模块恢复需求迭代

建议顺序：

1. 历史事件
2. 飞行记录详情
3. 设备信息 / 设备运行监控
4. 威胁评估
5. 预案策略配置
6. 系统管理与字典管理

每个模块都应：

1. 改动后立即构建
2. 进入实际页面检查
3. 截图确认
4. 再进入下一模块

## 7. 如果必须继续从当前仓库出发

如果暂时无法找到更好的中文基线，那么下一位接手者应当采用以下保守策略：

1. 先停止做业务功能新增
2. 只做“基础文案层修复专项”
3. 一次只修一个基础层文件
4. 每修一个文件就构建
5. 先修菜单、面包屑、标签页相关标题
6. 修完再恢复业务需求推进
